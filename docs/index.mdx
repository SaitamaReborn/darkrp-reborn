
## Deux façons de créer des addons — et même de les vendre

- **Lua (compatible GMod)** : des dossiers de scripts déposés sur votre serveur, rechargeables à chaud — idéal pour migrer l'existant et scripter vite. C'est la moitié « venir de Garry's Mod » du SDK.
- **C# (natif s&box)** : le [Dev Kit](/addons/csharp-demarrage) simule un serveur DarkRP Reborn dans l'éditeur ; vous codez contre le contrat d'API public (`Reborn`, `RebornEvents`, `IRebornPlayer`), et l'addon est intégré au jeu après revue. Les deux mondes voient les mêmes événements du jeu.

## Comment ça marche

Vos scripts Lua GMod tournent tels quels, syntaxe GLua comprise, avec l'API que vous connaissez déjà : `hook.Add`, `timer.Create`, `player.GetAll`, `ply:Nick()`, `DarkRP.createJob`…

Vos addons sont de simples dossiers de fichiers `.lua` déposés sur votre serveur, **aucune compilation, aucune republication** :

```text
data/rebornstudio/cityliferp/addons/
└── mon-addon/
    ├── addon.json      (manifeste, optionnel)
    └── main.lua        (point d'entrée)
```

```lua
-- main.lua, un premier addon en 6 lignes
hook.Add("PlayerInitialSpawn", "hello", function(ply)
	timer.Simple(8, function()
		if IsValid(ply) then ply:ChatPrint("Bienvenue sur mon serveur !") end
	end)
end)
```

Rechargez à chaud avec `addon_reload` dans la console serveur. Un addon en erreur est isolé et loggé, **il ne fait jamais tomber le serveur**.

## L'esprit du projet

- **Vos addons, votre serveur.** Le SDK n'impose aucune règle de gameplay : créez les métiers et systèmes que VOUS voulez.
- **Sécurité d'abord.** Chaque addon tourne dans un bac à sable isolé (pas d'accès disque hors de son dossier, budget CPU par appel, API curée).
- **Compatibilité mesurée, pas promise.** Chaque API est validée sur de vrais addons publics, [la matrice complète est publique](/compat/darkrp).
