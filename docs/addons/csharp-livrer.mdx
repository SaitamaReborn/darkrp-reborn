title: "Addons C# : soumettre son addon"
description: "Le circuit de publication : revue, intégration au jeu, déploiement. Ce qui est accepté, ce qui ne l'est pas."
---

Contrairement aux addons Lua (déposés sur un serveur), les addons C# sont
**intégrés au jeu lui-même** après revue — le modèle « marketplace curée » : votre
code tourne nativement, au même niveau que les systèmes du gamemode.

## Le circuit

1. **Développez** dans le Dev Kit (`devkit/code/MyAddons/<votre-ident>/`).
2. **Ouvrez une Pull Request** sur le [dépôt public darkrp-reborn](https://github.com/SaitamaReborn/darkrp-reborn) :
   fork, copiez votre dossier dans `addons/<votre-ident>/` avec sa fiche
   `addon.md`, le template de PR vous guide. (Pas de compte GitHub ? Un zip à
   l'équipe sur Discord marche aussi.)
3. **Revue** : lecture du code (checklist ci-dessous), test dans le Dev Kit puis
   sur le serveur de dev — directement dans la PR.
4. **Intégration** : l'addon est posé tel quel dans `code/Addons/<votre-ident>/`
   du jeu et part en production au publish suivant. Votre `Author` s'affiche dans
   `csaddon_list`.
5. **Mises à jour** : renvoyez le dossier avec `Version` incrémentée ; même
   circuit, plus rapide.

## Checklist de revue

Accepté :

- Ne référence **que** le contrat public (`RebornAddon`, `Reborn`, `RebornEvents`,
  `IRebornPlayer`) et l'API s&box standard.
- Logique serveur : argent via `GiveMoney`/`TakeMoney`, jamais de valeur inventée
  côté client.
- `OnUpdate`/`OnFixedUpdate` sobres : pas d'allocations par frame, pas de scan de
  scène à chaque tick (mettez en cache, utilisez `Reborn.Every`).
- Textes joueurs propres (pas d'insulte, pas de marque tierce), idéalement
  localisés via `p.Language`.
- Un seul addon par dossier, ident stable, `Version` renseignée.

Refusé d'office :

- Patch ou copie modifiée des fichiers du contrat (`code/Addons/*.cs`).
- Réflexion sur les types internes du jeu, accès fichiers hors
  `Reborn.SaveData`/`LoadData`, requêtes HTTP.
- Économie déséquilibrée (impression d'argent, boucles de récompense infinies) —
  l'équilibrage fait partie de la revue.
- Code obfusqué ou illisible : la revue lit tout.

## Qui possède quoi

Votre addon reste **votre code** : vous le signez (`Author`), vous décidez de ses
mises à jour. En l'envoyant, vous autorisez DarkRP Reborn à l'embarquer dans le
jeu publié et à le désactiver si un problème de prod l'exige. Le contrat d'API
est versionné : si un changement cassant arrive un jour, l'équipe vous prévient
avec la migration.

## Vendre son addon

Le système est pensé pour que vous puissiez **créer — et même vendre — vos
addons** sans jamais avoir accès au code source du jeu ni à ses bases de
données (le contrat vous isole par design : pas d'accès BDD, persistance
sandboxée par addon via `SaveData`).

- **Gratuit / communautaire** : le circuit ci-dessus (PR publique), votre nom
  crédité en jeu.
- **Payant** : votre code reste le vôtre — vendez-le en direct (votre boutique,
  Discord) aux propriétaires de serveurs ou à l'équipe. Deux canaux techniques,
  à choisir selon la protection voulue :
  - **Addon Lua** : déposé sur UN serveur (`data/addons/`), sa logique serveur
    n'est **jamais distribuée aux joueurs** — c'est le bon canal pour du code
    payant que vous voulez protéger.
  - **Addon C#** : intégré au jeu publié — sur s&box, le source d'un package
    publié est techniquement extractible par les joueurs. Recommandé pour les
    addons gratuits/crédités, ou vendez une **licence d'utilisation** plutôt
    qu'un secret de code.
- Un espace de mise en avant des addons payants est à l'étude ; en attendant,
  contactez l'équipe pour être référencé.

<Note>
Vous voulez plutôt un script rapide, rechargeable à chaud sur un serveur, sans
revue ? C'est le rôle des [addons Lua](/addons/demarrage) — les deux mondes
voient les mêmes événements du jeu.
</Note>
