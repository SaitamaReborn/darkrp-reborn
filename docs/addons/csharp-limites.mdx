title: "Addons C# : limites du système (v2)"
description: "Le contrat v2 a levé la plupart des limites v1. Voici ce qui reste fermé, et pourquoi."
---

Le contrat **v2** a ouvert la majorité de ce qui manquait en v1 : téléportation,
vie, changement de métier, items, argent fixé, spawn de props, et surtout
**création de métiers et d'items au runtime** (ils apparaissent dans le F4 et
l'inventaire comme les natifs). Il reste quelques limites — chacune a une vraie
raison.

## Ouvert en v2 ✓

| Avant (v1) | Maintenant (v2) |
|---|---|
| Pas de création de métiers | `Reborn.RegisterJob( new RebornJobSpec {...} )` → visible dans le F4 |
| Pas de création d'items | `Reborn.RegisterItem( new RebornItemSpec {...} )` + `p.GiveItem( id, n )` |
| Pas de téléportation / vie / métier / argent fixé | `p.Teleport()`, `p.SetHealth()`, `p.TrySetJob()`, `p.SetMoney()` + `p.Position`, `p.Health` |
| Pas de spawn de props via le contrat | `Reborn.SpawnProp( model, pos, yaw, Ident )` |

`TrySetJob` respecte les règles normales du jeu (postes complets, métiers à
vote) : un addon ne contourne pas un vote de la ville.

## Ce qui reste fermé, et pourquoi

| Quoi | Pourquoi |
|---|---|
| Bloquer / modifier le chat | Le chat est un broadcast direct pair-à-pair (choix d'architecture éprouvé sur serveur dédié) : le message est déjà chez tout le monde quand le serveur le voit. Un veto exigerait de réarchitecturer le chat — à l'étude, pas pris à la légère. |
| Annuler un événement | Les événements sont des notifications (pas de « prevent default »). Le veto chat/dégâts fait partie du même chantier. |
| UI custom / menus / HUD côté client | Les addons tournent côté serveur uniquement. L'UI client d'addon est LE chantier v3. En attendant : `Notify`, `Chat`, `Broadcast`. |
| Requêtes HTTP | Refusé en revue — et de toute façon gaté par la whitelist HTTP du moteur. Un besoin légitime (webhook Discord…) peut être whitelisté au cas par cas : demandez. |
| Accès aux bases de données du jeu | **Promesse de sécurité du système**, pas un manque : le contrat ne les exposera jamais. Votre persistance = `SaveData`/`LoadData`. |
| Réflexion sur les types internes / classes privées du jeu | Contrat public + API s&box standard uniquement (critère n°1 de la revue). |
| Modifier les fichiers du contrat | Refusé d'office. |
| Masquer une commande du jeu | Priorité fixe : natif > addons Lua > addons C#. |
| Imprimer de l'argent / boucles de récompense infinies | Refusé en revue (équilibrage). |

## Nuances

- **Le monde est réel côté serveur, simulé dans le Dev Kit.** `SpawnProp`
  retourne `null` et `p.GameObject` est `null` dans le Dev Kit (pas de monde en
  v2) — la console simulée loggue l'action. Testez la partie monde sur le
  serveur de dev, gardez les tests `!= null`.
- **Le code C# intégré au jeu publié est techniquement extractible** par les
  joueurs (réalité s&box). Pour du code à protéger ou à vendre, le canal Lua
  serveur est le bon choix — voir
  [Vendre son addon](/addons/csharp-livrer#vendre-son-addon).
- **Le contrat est versionné** (`Reborn.ApiVersion`, actuellement 2) : toute
  évolution cassante serait annoncée avec sa migration.

## Le chantier v3

Dev Kit. Dites-nous ce qui vous bloque en premier.
