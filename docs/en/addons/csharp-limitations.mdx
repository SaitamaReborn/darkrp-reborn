title: "C# addons: system limitations (v3)"
description: "Contract v3 opens UI, HTTP and storage. Here is the little that stays closed, and why."
---

Contract **v3** opens nearly everything: on top of the v2 powers (teleport,
health, jobs, items, money, props, runtime job/item creation), addons now have
**their own UI** (windowed menus + HUD cards, game design by default and
customizable), **HTTP** (domain allowlisted at integration) and **their own
keyed storage**. The little that stays closed has a real reason.

## Opened in v2 and v3 ✓

| Before (v1) | Now (v2) |
|---|---|
| No job creation | `Reborn.RegisterJob( new RebornJobSpec {...} )` → shows in the F4 menu |
| No item creation | `Reborn.RegisterItem( new RebornItemSpec {...} )` + `p.GiveItem( id, n )` |
| No teleport / health / job / exact money | `p.Teleport()`, `p.SetHealth()`, `p.TrySetJob()`, `p.SetMoney()` + `p.Position`, `p.Health` |
| No prop spawning through the contract | `Reborn.SpawnProp( model, pos, yaw, Ident )` |
| No custom UI | **v3**: `RebornMenu` (windows, tables, inputs, buttons) + `ShowHudCard` — game design by default, `Accent`/`Background` overridable |
| No HTTP | **v3**: `Reborn.HttpGetAsync`/`HttpPostJsonAsync` — domain allowlisted at integration |
| One save blob per addon | **v3**: `SaveData( Ident, key, json )` — your own keyed database |

`TrySetJob` follows the game's normal rules (full jobs, vote-gated jobs): an
addon cannot bypass a city vote.

## What stays closed, and why

| Thing | Why |
|---|---|
| Block / modify chat | Chat is a direct peer-to-peer broadcast (a battle-tested dedicated-server design): the message has reached everyone by the time the server sees it. A veto requires re-architecting chat — under consideration, not taken lightly. |
| Cancel any event | Events are notifications (no "prevent default"). Chat/damage veto belongs to the same work. |
| Arbitrary client-side razor/HTML | Addon UI goes through the `RebornMenu` framework (rich widgets, consistent design) — addon-shipped arbitrary razor stays closed (security + visual coherence). Missing a widget? Ask, the framework grows. |
| Access to the game's databases | A **security promise of the system**, not a gap: it is the "create without access to our databases" pitch sold to creators. Every addon has ITS OWN keyed database (`SaveData`). |
| Reflection into internals / private game classes | Public contract + standard s&box API only (review criterion #1). |
| Modifying the contract files | Rejected outright. |
| Shadowing a built-in command | Fixed priority: native > Lua addons > C# addons. |
| Money printing / infinite reward loops | Rejected in review (balance). |

## Nuances

- **The world is real on the server, simulated in the Dev Kit.** `SpawnProp`
  returns `null` and `p.GameObject` is `null` in the Dev Kit (no world in v2) —
  the simulated console logs the action instead. Test world code on the dev
  server, keep the `!= null` checks.
- **C# code integrated into the published game is technically extractable** by
  players (s&box reality). For code you want to protect or sell, the
  server-side Lua channel is the right choice — see
  [Selling your addon](/en/addons/csharp-shipping#selling-your-addon).
- **The contract is versioned** (`Reborn.ApiVersion`, currently 3): any breaking
  change would be announced with its migration.
